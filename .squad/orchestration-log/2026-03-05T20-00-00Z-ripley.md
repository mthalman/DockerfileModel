## Decisions Produced (8 total)

1. **Use Lean 4 Built-in `Parsec`** — Not third-party library. Zero external dependencies via `Lean.Data.Parsec` or `Std.Internal.Parsec.String`.
2. **Parser Output Type** — Produce existing `Token` directly, not intermediate AST.
3. **Module Structure** — Flat under `lean/DockerfileModel/Parser/`, one file per concern (Basic.lean, Tokens.lean, From.lean, Arg.lean, etc.).
4. **Combinator Mapping** — Systematic translation: Sprache `from...in...select` → Lean `do` notation; `.Or()` → `attempt p <|> q`; `.XOr()` → `p <|> q`.
5. **Hard Parts Handling** — Thread escape char as `ParserConfig` parameter; whitespace/line-continuation/quoting machinery in `Basic.lean`; variable refs with `attempt` backtracking.
6. **Round-Trip Theorem Strategy** — Bottom-up per-combinator proofs. Prove FROM first, extract reusable lemmas, extend to simpler instructions.
7. **Implementation Order** — Start with Basic.lean + Tokens.lean, then From.lean with concurrent proofs, then simpler instructions.
8. **Lakefile Changes** — No new dependencies. `lean_lib` auto-discovers modules in subdirectories.

## Key Insights
- Sprache's LINQ syntax maps perfectly to Lean's `do` notation with `<-`.
- The hardest work is whitespace/line-continuation/quoting machinery in `Basic.lean` — get that right and per-instruction parsers are mechanical.
- Full round-trip proof for all 18 instructions is large; scope Phase 2 to FROM + 3-4 simple instructions.
